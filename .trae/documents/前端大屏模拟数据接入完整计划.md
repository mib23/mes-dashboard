# 前端大屏模拟数据接入完整计划

## Summary

- 目标：为现有 4 个大屏页面补齐一套可持续运行的“高拟真动态模拟数据接入方案”，让演示项目在未接入真实 MES/ERP/IoT 的前提下，也能持续输出接近真实生产环境的数据变化与告警联动。
- 范围：覆盖 `frontend/src/pages/FactoryOverview.tsx`、`frontend/src/pages/KnittingWorkshop.tsx`、`frontend/src/pages/LinkingSewing.tsx`、`frontend/src/pages/FinishingPackaging.tsx` 四个页面。
- 方案方向：采用“前端内本地 Mock WebSocket + 统一数据模型 + Zustand 全局状态”的结构，优先保证后续可平滑切换为真实 WebSocket / API。
- 成功标准：
  - 四个大屏页面全部由统一数据源驱动，不再直接依赖写死 JSX 文案数字。
  - 核心指标可按秒级自动刷新，且指标之间具备业务联动，不是简单随机数跳动。
  - 具备“连接中 / 在线 / 离线 / 重连”状态表现，接口层可替换为真实服务。
  - 页面长期运行时不会因定时器、订阅、图表实例未清理而明显泄漏。

## Current State Analysis

### 仓库现状

- 项目当前为纯前端仓库，业务代码位于 `frontend/`，根目录未发现后端服务目录。
- 技术栈已确定为 `React + TypeScript + Vite + ECharts + Zustand`，见 `frontend/package.json`。
- 页面路由由 `frontend/src/App.tsx` 管理，当前已挂载四个大屏页面。
- `frontend/src/main.tsx` 仅负责挂载 `App`，当前没有 Provider、全局状态初始化或实时数据连接初始化逻辑。
- `frontend/src/` 当前只有 `pages/`、`assets/` 和基础样式文件，没有 `services/`、`store/`、`types/`、`mock/`、`hooks/` 等数据接入层目录。

### 页面现状

- `frontend/src/pages/FactoryOverview.tsx`
  - 为纯静态展示页。
  - 头部时间、系统状态、产量达成率、设备状态、急单列表、OEE 排名等全部硬编码。
  - 没有任何 `useEffect`、订阅逻辑、接口调用或状态来源。
- `frontend/src/pages/KnittingWorkshop.tsx`
  - 所有 OEE 相关指标、产量进度、故障告警、机台矩阵均为静态内容。
  - 页面中已有较明确的业务模块分区，适合直接替换为数据驱动组件。
- `frontend/src/pages/LinkingSewing.tsx`
  - 班组效率、WIP 瓶颈、质量警示、员工排行榜、底部汇总卡片全部为写死数据。
  - WIP 瓶颈区域有明确的业务语义，适合做“工序联动 + 瓶颈高亮”模拟。
- `frontend/src/pages/FinishingPackaging.tsx`
  - 洗水/烘干设备状态、批次流转、齐套率、发货状态等均为静态。
  - 页面天然适合模拟“设备运行 -> 批次推进 -> 齐套率变化 -> 发货状态变化”的链路。

### 文档约束

- `docs/PRD.md` 已定义四个页面的业务目标、指标口径与实时性要求。
- `docs/Technical_Architecture.md` 已给出未来目标架构：前端通过 WebSocket 订阅后端推送，后端再聚合 MES / ERP / IoT 数据。
- 当前前端实现与目标架构之间缺少一层“可替换的数据访问抽象”，这是本次计划需要优先补齐的核心。

## Assumptions & Decisions

- 决策：本阶段不新增真实后端，不改为多仓，不引入数据库；所有模拟逻辑先在前端仓库内闭环实现。
- 决策：使用“本地 Mock WebSocket 客户端”模拟真实推送语义，而不是直接在页面中 `setInterval` 随机改数。
- 决策：建立统一数据模型与实时订阅接口，使未来接入真实 `socket.io-client` 时只替换 adapter 层。
- 决策：高拟真优先于快速随机刷新，指标变化必须具备因果关系。
- 决策：四个大屏共享同一“工厂运行时世界状态”，不同页面读取其各自切片数据，避免四页各自独立随机。
- 假设：当前仅需本地演示可运行，不要求与已有权限体系、登录体系或后端联调。
- 假设：现阶段可以增补必要前端依赖；若需要对齐未来架构，可补充 `socket.io-client`，但 Mock 首版也可通过自定义事件总线实现统一接口。
- 假设：页面当前样式结构整体保留，以“替换数据来源 + 补少量状态显示”为主，不重做 UI。

## Proposed Changes

### 1. 建立统一领域类型与屏幕数据 Schema

#### 新增文件

- `frontend/src/types/dashboard.ts`
- `frontend/src/types/realtime.ts`

#### 内容与做法

- 为四个页面抽出统一类型定义，至少包括：
  - 全局元信息：当前班次、系统时钟、连接状态、刷新时间。
  - 工厂总览：计划/实际产量、达成率、质量良率、设备开机率、急单列表、车间排行榜。
  - 编织车间：OEE、Availability、Performance、Quality、织片进度、机台状态、活跃故障。
  - 套口缝合：班组效率、工序 WIP、瓶颈工序、质量缺陷、员工排行榜、考勤与机器汇总。
  - 后整包装：洗水/烘干设备、批次流、齐套矩阵、出货 Dock 状态。
- 补充实时层协议类型：
  - `ConnectionStatus`
  - `RealtimeChannel`
  - `RealtimeMessage<T>`
  - `DashboardSnapshot`
  - `DashboardDelta`
- 要求类型命名与未来真实接口保持一致，后续真实数据接入时优先复用。

#### 目的

- 消除页面内散落的“隐式数据结构”。
- 为 Zustand store、mock engine、页面组件之间建立稳定契约。

### 2. 建立模拟数据种子与业务规则引擎

#### 新增文件

- `frontend/src/mock/seeds/factorySeed.ts`
- `frontend/src/mock/seeds/knittingSeed.ts`
- `frontend/src/mock/seeds/linkingSeed.ts`
- `frontend/src/mock/seeds/finishingSeed.ts`
- `frontend/src/mock/engine/createWorldState.ts`
- `frontend/src/mock/engine/tickEngine.ts`
- `frontend/src/mock/engine/rules.ts`
- `frontend/src/mock/engine/random.ts`

#### 内容与做法

- 抽象一个“工厂世界状态（world state）”，作为所有页面共享的唯一真相源。
- `seed` 负责初始化静态基础盘：
  - 订单池、班次信息、机台列表、工序列表、员工列表、设备台账、批次池。
- `tickEngine` 按固定节奏推进状态：
  - 推荐每 1 秒推进一次世界时钟。
  - 每 3-5 秒触发一次可见业务变化。
  - 每 15-30 秒注入一次异常事件候选。
- `rules.ts` 定义联动规则，至少包括：
  - 设备停机或待机会拉低编织产量增速与全厂达成率。
  - 编织端产出波动会影响后续套口/缝合的 WIP 堆积。
  - 某工序积压升高时，瓶颈工序自动切换并触发红色高亮。
  - 质量缺陷频发会拉低良率与班组效率，并可能触发告警。
  - 后整洗水设备故障会拖慢批次推进、降低订单齐套率和发货活跃度。
- 引入随机但受控的扰动策略：
  - 小幅波动围绕业务基线变化，避免大起大落不真实。
  - 异常事件有冷却时间，避免页面始终告警刷屏。
  - 指标变化要有上下限、恢复曲线和缓冲惯性。

#### 目的

- 让模拟不只是“随机数字”，而是“有业务逻辑的动态生产链路”。

### 3. 建立可替换的实时接入层

#### 新增文件

- `frontend/src/services/realtime/realtimeClient.ts`
- `frontend/src/services/realtime/mockSocketClient.ts`
- `frontend/src/services/realtime/liveSocketClient.ts`
- `frontend/src/config/runtime.ts`

#### 内容与做法

- 定义统一实时客户端接口，例如：
  - `connect()`
  - `disconnect()`
  - `subscribe(channel, handler)`
  - `getSnapshot()`
  - `onStatusChange(handler)`
- `mockSocketClient.ts`
  - 基于前端本地 engine 实现“类 WebSocket 推送”。
  - 支持频道划分：`factory-overview`、`knitting`、`linking-sewing`、`finishing-packaging`、`global-status`。
  - 推送全量快照 + 增量事件。
  - 模拟连接建立、重连、短暂断流、恢复后的补快照。
- `liveSocketClient.ts`
  - 先实现为空壳适配器或最小占位版本。
  - 约定未来接真实 `socket.io-client` 的入口与事件名，不在本阶段启用。
- `runtime.ts`
  - 提供运行模式开关，例如 `mock-realtime` / `live-realtime`。
  - 首版默认固定为 `mock-realtime`。

#### 目的

- 让页面与“数据来自哪里”彻底解耦。
- 为真实接口落地预留最小替换面。

### 4. 建立全局状态层与订阅编排

#### 新增文件

- `frontend/src/store/useDashboardStore.ts`
- `frontend/src/hooks/useRealtimeDashboard.ts`
- `frontend/src/hooks/useConnectionStatus.ts`

#### 内容与做法

- 使用 Zustand 存储四屏共享数据、连接状态、最新更新时间和告警计数。
- `useRealtimeDashboard.ts`
  - 在应用启动时初始化 realtime client。
  - 订阅各频道并将消息规整写入 store。
  - 统一负责连接、断开、重连与清理。
- `useConnectionStatus.ts`
  - 为页面头部提供统一的连接状态文案和颜色映射。
- Store 设计要求：
  - 支持 `replaceSnapshot` 和 `applyDelta` 两类写入。
  - 支持页面按切片订阅，避免四屏都因全局任意字段变化而重渲染。
  - 保留 `source: 'mock' | 'live'` 标记，便于未来切换。

#### 目的

- 解决当前项目完全没有数据层的问题。
- 控制渲染范围，减少大屏长期运行的性能风险。

#### 目的

## 业务联动设计明细

### 工厂总览

- 今日计划与实际产量来自全厂汇总。
- 达成率受编织、套口缝合、后整三个车间总吞吐影响。
- 活跃设备数与停机数来自各车间设备状态聚合。
- 急单列表从订单池里筛选优先级高、延迟风险高的订单。
- OEE / 效能排行榜由各车间衍生指标实时重排。

### 编织车间

- 每台机台有状态机：`running / standby / warning / downtime / maintenance`。
- `running` 机台按效率产出织片；`downtime` 会降低产出。
- 故障类型至少覆盖断纱、漏针、保养提醒。
- OEE 三要素由机台状态和质量波动共同计算。

### 套口与缝合

- 上游供给不足时，某些工序 WIP 降低；供给过量或工序故障时，WIP 堆积。
- 瓶颈工序始终取当前最高滞留值，颜色和提醒随之切换。
- 质量缺陷频发会反向影响班组效率。
- 员工排行榜受工位状态和缺陷率影响，可发生名次变化。

### 后整包装

- 洗水 / 烘干设备进度推进决定批次完成速度。
- 设备故障时对应批次 ETA 延长。
- 批次完成后推进到齐套统计，再影响发货 Dock 活跃状态。
- 若某订单尺码或颜色未齐套，则订单状态保持非完成。

## Implementation Steps

### 阶段 1：搭数据骨架

1. 新建 `types/`，定义统一 schema、事件消息、连接状态。
2. 新建 `mock/seeds/`，整理四屏需要的基础实体。
3. 新建 `mock/engine/`，先实现世界状态、tick 机制和基础计算。

### 阶段 2：搭实时接入层

1. 抽象 `realtimeClient` 接口。
2. 实现 `mockSocketClient`，打通连接、订阅、推送、重连模拟。
3. 预留 `liveSocketClient` 占位，约定真实接入替换点。

### 阶段 3：搭状态层

1. 新建 Zustand store。
2. 实现 `replaceSnapshot` / `applyDelta`。
3. 新建全局订阅 Hook，确保应用生命周期内只有一个连接实例。

### 阶段 4：逐页替换

1. 先改 `FactoryOverview.tsx`，验证全局汇总数据链路。
2. 再改 `KnittingWorkshop.tsx`，验证设备状态、告警、产量联动。
3. 再改 `LinkingSewing.tsx`，验证 WIP 瓶颈与排行榜。
4. 最后改 `FinishingPackaging.tsx`，验证设备进度与订单齐套链路。

### 阶段 5：补稳定性与演示体验

1. 增加连接状态提示与断线恢复逻辑。
2. 补平滑更新、滚动列表稳定性、定时器清理。
3. 补必要的工具函数与重复 UI 抽象。

## Verification Steps

### 功能验证

- 进入四个页面后，所有主要数值、列表、状态卡片都来自统一数据源并持续刷新。
- 刷新节奏满足“核心指标秒级变化、列表和告警有节奏更新”的预期。
- 编织故障出现后，可观察到：
  - 编织产量增速下降；
  - 工厂总览达成率受影响；
  - 下游部分工序 WIP 或后整批次节奏出现联动变化。
- 套口与缝合页面中，瓶颈工序会随 WIP 变化自动切换。
- 后整页面中，设备告警会拉长批次 ETA，并影响齐套与发货状态。

### 技术验证

- 切换路由时不会重复创建多个 realtime 连接实例。
- 页面卸载或热更新后，定时器与订阅能正确清理。
- mock 断流时页面保留最近快照，恢复后可自动继续更新。
- Store 切片订阅后，页面不会因无关数据更新发生明显整页抖动。

### 构建与质量验证

- 执行 `npm run build` 可通过。
- 执行 `npm run lint` 不引入新的 ESLint 报错。
- 对本次新增或重构的关键文件执行诊断，确保没有类型错误。

## Risks

- 风险：若直接在页面内部写定时器，后续很难切真实接口。
  - 应对：统一经由 realtime adapter + store。
- 风险：高拟真规则过多会让首版实现复杂度失控。
  - 应对：先实现核心因果链，再逐步增加细节扰动。
- 风险：四个页面若都订阅全量状态，渲染压力会偏高。
  - 应对：store 切片订阅 + 页面内数组渲染优化。
- 风险：断线演示若处理不当会引发空白闪烁。
  - 应对：保留最近快照并只切换连接状态标签。

## Out Of Scope

- 不接入真实 MES/ERP/IoT 接口。
- 不建设后端 NestJS、MySQL、Redis 服务。
- 不在本阶段重做整套页面视觉或路由结构。
- 不处理权限、登录、菜单系统等非大屏演示需求。

- 页面数据表现接近真实生产节奏，具备故障、恢复、瓶颈、排行榜变化等演示价值。
- 代码结构具备下一阶段接入真实 WebSocket / API 的清晰替换路径。
